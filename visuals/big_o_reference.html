The Golden Rules

Drop constants

O(2n) → O(n) · O(500) → O(1)
Constants don't affect growth rate — only the dominant term matters at scale.

Drop lower-order terms

O(n² + n) → O(n²) · O(n log n + n) → O(n log n)
As n grows, the smaller term becomes negligible.

Sequential → add

Two separate loops = O(n) + O(n) = O(2n) → O(n)
Not O(n²) — nesting multiplies, sequencing adds.

Nested → multiply

Loop inside a loop = O(n) × O(n) = O(n²)
Each outer step triggers a full inner traversal.

Halving → log n

Any time each step cuts the problem in half: O(log n)
After k halvings, n/2ᵏ = 1 → k = log₂(n)

Recursion trees

Count levels (log n) × work per level (n) = O(n log n)
Or levels × branching factor for exponential cases.

What Code Structure → What Complexity

O(1) — Constant

No loops. Direct access or fixed computation.

def get_first(arr):
    return arr[0]   # one operation

def lookup(d, key):
    return d[key]   # hash table lookup

O(log n) — Logarithmic

Problem halves each iteration. Classic: binary search.

while lo <= hi:
    mid = (lo + hi) // 2
    if arr[mid] == target:
        return mid
    elif arr[mid] < target:
        lo = mid + 1  # discard left half
    else:
        hi = mid - 1  # discard right half

O(√n) — Square root

Loop runs up to √n times — checking divisors only needs to go to √n.

def is_prime(n):
    for i in range(2, int(n**0.5) + 1):
        if n % i == 0:
            return False
    return True

O(n) — Linear

Single loop over all elements. Work scales directly with n.

def find_max(arr):
    best = arr[0]
    for x in arr:    # n iterations
        if x > best:
            best = x
    return best

O(n) — Two separate loops

O(n) + O(n) = O(2n) = O(n). Sequential, not nested!

for x in arr:      # first pass O(n)
    process(x)

for x in arr:      # second pass O(n)
    do_other(x)
# Total: O(n) — NOT O(n²)

O(n log n) — Linearithmic

Sorting, or divide-and-conquer where each level does O(n) work.

def merge_sort(arr):
    if len(arr) <= 1: return arr
    mid = len(arr) // 2
    L = merge_sort(arr[:mid])   # log n levels
    R = merge_sort(arr[mid:])
    return merge(L, R)           # O(n) per level

O(n²) — Quadratic

Nested loops, each going to n. O(n) × O(n) = O(n²).

for i in range(n):      # outer: n steps
    for j in range(n):  # inner: n steps
        print(arr[i], arr[j])
# n × n = n² total operations

O(n²) — Even if inner doesn't reach n

Inner loop starting at i+1 still = n(n-1)/2 ≈ n²/2 → O(n²)

for i in range(n):
    for j in range(i+1, n): # still O(n²)!
        # pairs: n(n-1)/2 ≈ n²/2
        check_pair(arr[i], arr[j])

O(n³) — Cubic

Three nested loops. Rare but real (e.g. Floyd-Warshall).

for i in range(n):
    for j in range(n):
        for k in range(n):
            dp[i][j] = min(
                dp[i][j],
                dp[i][k] + dp[k][j])

O(2ⁿ) — Exponential

Two recursive calls each time. Branches double every level.

def fib(n):             # naive recursion
    if n <= 1: return n
    return fib(n-1) + fib(n-2)
# Each call → 2 more calls → tree of 2ⁿ nodes
# Fix: memoisation → O(n)

O(n!) — Factorial

Generating all permutations. n choices, then n-1, then n-2...

def permutations(arr, current=[]):
    if not arr:
        results.append(current)
        return
    for i, x in enumerate(arr):
        permutations(    # n × (n-1) × ... = n!
            arr[:i] + arr[i+1:],
            current + [x])

O(n) — Looks like O(n²) but isn't

Hash set lookup is O(1), not O(n). The key insight Hugo will probe!

def two_sum(nums, target):
    seen = {}           # hash table
    for num in nums:   # O(n) outer loop
        complement = target - num
        if complement in seen:  # O(1)!
            return [seen[complement], ...]
        seen[num] = ...
# Total: O(n) × O(1) = O(n)
💡 The 5-step method: (1) count loops and their ranges, (2) check if loops are nested or sequential, (3) check if any loop variable halves/grows exponentially, (4) check inner data structure operations (dict/set = O(1), list `in` = O(n)), (5) combine: sequential adds, nested multiplies.

Why Halving Gives O(log n)

Each step eliminates half the remaining candidates. After k steps, you have n/2ᵏ elements left. You stop when that equals 1, so k = log₂(n).

Other O(log n) Patterns

Balanced BST lookup

Each comparison eliminates half the tree. Height of balanced BST = log n.

def bst_search(node, target):
    if not node: return None
    if node.val == target: return node
    if target < node.val:
        return bst_search(node.left, target)
    return bst_search(node.right, target)

Integer division loop

Any loop where n is divided by a constant each iteration = O(log n).

n = 1024
steps = 0
while n > 1:
    n //= 2    # halves each time
    steps += 1
# steps = log₂(1024) = 10

Watch out: O(n log n) sorting

If you sort inside an O(n) loop, total is O(n log n) × O(n) = O(n² log n). Always sort before the loop.

# BAD: O(n² log n)
for i in range(n):
    arr.sort()     # O(n log n) inside loop!
    process(arr[i])

# GOOD: O(n log n)
arr.sort()         # sort once outside
for i in range(n):
    process(arr[i])

Why Merge Sort is O(n log n)

Two independent reasons combine: the recursion tree has log n levels, and each level does exactly n units of merge work. Total = n × log n.

Array size (n) n = 8

Why log n levels?

Each level halves the subarray size. Starting from n, we reach size 1 after log₂(n) splits. This is the same halving logic as O(log n) — just applied to the recursion depth.

Why n work per level?

At every level, we're merging all elements exactly once. Level 1: one merge of n elements. Level 2: two merges of n/2 each = n total. Level 3: four merges of n/4 each = n total. Always n.

The multiplication

log n levels × n work per level = n log n total operations. This is why sorting is the "best you can do" for comparison-based sorts — you can't do better than O(n log n).

Space complexity

Merge sort needs O(n) extra space — the merge step creates auxiliary arrays. Quicksort avoids this (in-place) but has O(n²) worst case. Heapsort is O(1) space and O(n log n) guaranteed.

O(n log n) vs O(n²): The Practical Gap

At n=10,000: merge sort ≈ 132,000 ops. Bubble sort = 100,000,000 ops. That's a 756× difference.
